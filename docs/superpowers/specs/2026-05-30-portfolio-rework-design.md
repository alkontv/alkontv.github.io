## 4. Архитектурные решения (сквозные)

### 4.1 i18n — самописный словарь (подход A)

Новый модуль `app/i18n/`:

```
app/i18n/
  langStore.ts     // Zustand+persist: lang:'ru'|'en', setLang, toggleLang
  content/
    en.ts          // словарь EN — задаёт ТИП Content (источник формы)
    ru.ts          // словарь RU: const ru: Content = {...} (гарантия ключей)
    index.ts       // export type Content; export const dict = { ru, en }
  useT.ts          // хук: const t = useT(); -> dict[lang]
```

- `langStore` копирует проверенный паттерн `themeStore` (Zustand + `persist`
  + `partialize`), хранит выбранный язык в localStorage. Дефолт — `en`
  (как сейчас в Notion; международная аудитория).
- `en.ts` определяет тип `Content` (через `satisfies`/экспорт типа), `ru.ts`
  типизируется как `Content` — TypeScript заставит держать одинаковый
  набор ключей в обоих языках. Это страховка от «забыл перевести».
- `useT()` — селектор `dict[useLangStore(s => s.lang)]`; перерисовка только
  при смене языка (гранулярный селектор, как у тем).
- Локализуемый текст 3D-сцены: `<Text>{t.hero.title}</Text>` — troika
  перерисует глиф при смене строки автоматически.

**Почему не библиотека:** одна страница, нет роутинга, `output: export`,
малый объём текста, состояние уже на Zustand. next-intl/react-i18next —
лишняя масса и второй подход к состоянию.

### 4.2 Контент-модель: единый источник правды

Сейчас контент непоследователен: `constants/{projects,work,footer}.ts`
вынесены, а имя/слоганы/мета — захардкожены в JSX (`hero/index.tsx:38`,
`hero/TextWindow.tsx`, `layout.tsx`, `AwwardsBadge.tsx`). Сводим **весь**
человекочитаемый текст в `app/i18n/content/`:

- **Чисто текстовые строки** (hero title, слоганы окна, заголовки секций,
  подписи футера, CTA, мета) — ключи в `content/{ru,en}.ts`.
- **Структурные данные с текстом** (`projects`, `work`) — остаются в
  `constants/` как массивы (ссылки, иконки, технологии, изображения —
  языконезависимы), а их **локализуемые поля** (`description`, `role`,
  `title` где нужно) делаем формы `{ ru: string; en: string }` и читаем
  по текущему языку. Технологии (`Kafka`, `Flutter`...) — общие, не
  переводятся.

Итог: «поменять имя/слоган/перевод» = правка одного `content`-файла; это
закрывает критическую находку аудита про разбросанный контент.

### 4.3 Шрифты и кириллица

3D-текст рисуется troika-шрифтами `public/soria-font.ttf` (заголовки,
широкий геометрический) и `public/Vercetti-Regular.woff` (акценты). Оба —
**латиница без кириллицы**, русские буквы в 3D не отрендерятся.

**Стратегия (сохраняет EN-вид нетронутым):** выбор шрифта зависит от
языка. Для `en` — текущие `soria`/`Vercetti`. Для `ru` — добавляем
кириллические аналоги, близкие по духу, и подменяем `font`-проп в `Text`
по `lang`.

Кандидаты (бесплатные, с кириллицей) — **финал на отдельном ревью с
превью**:
- Замена `soria` (широкий дисплейный): **Unbounded** (Google Fonts,
  геометрический, модный, есть кириллица).
- Замена `Vercetti` (моно-акцент): **JetBrains Mono** или **Martian Mono**
  (кириллица есть, инженерный вайб — даже уместнее для разработчика).

DOM-тексты (вне 3D) используют системные/Geist — кириллица там уже
поддержана, проблем нет.

### 4.4 Деплой — GitHub Pages

- **Репозиторий: `alkontv.github.io`** (user-site) → URL
  `https://alkontv.github.io`, **`basePath` пустой** (чистый корневой
  URL — лучше для личного бренда). [РЕШЕНИЕ ЗА ПОЛЬЗОВАТЕЛЯ — см. §8]
- **Ветка деплоя:** текущий workflow `.github/workflows/nextjs.yml`
  слушает `master`; у нас дефолт `main` → переключаем workflow на `main`.
- GA: до получения своего `NEXT_PUBLIC_GA_ID` — отключаем (не рендерим
  тег), мёртвый `env`-блок из `next.config.ts` убираем.
- Custom domain (`GH_PAGES_CUSTOM_DOMAIN`) — пока не задаём.

  optimization) — для статического портфолио это ок.
- Большие `.glb` в git-истории (baseline) — почистим из рабочего дерева в
  Фазе 3.
